# 大量文件渐进式加载优化方案

## 问题分析

当前方案（先扫描所有文件 → 分组 → 显示）在文件多的目录下：
- 用户等待太久（空白屏幕）
- 所有文件必须扫描完才能显示
- 用户体验差

## 官方推荐方案

### 1. 增量数据虚拟化 (ISupportIncrementalLoading)
- 边加载边显示
- 用户滚动时自动加载更多
- 滚动条实时反馈

### 2. StorageFileQueryResult 分批获取
- GetFilesAsync(index, count) 支持分批
- 先显示第一批，后台继续处理

### 3. x:Phase 渐进式渲染
- 先渲染关键元素（图片）
- 后渲染次要元素（格式按钮）

## 推荐方案：**渐进式显示 + 动态分组**

### 核心思路
```
1. 快速启动：先扫描 50-100 个文件，立即显示
2. 后台处理：继续扫描剩余文件
3. 动态分组：边扫描边合并重复格式
4. 平滑更新：UI 响应式刷新，无卡顿
```

### 详细步骤

| 步骤 | 说明 | 优先级 |
|------|------|--------|
| **步骤1** | 快速首屏：第一批 50 个文件立刻显示，不分组 | P0 |
| **步骤2** | 后台继续：扫描剩余文件，每 100 个触发一次 UI 更新 | P0 |
| **步骤3** | 动态分组：累计一定数量后，重新分组并更新 UI | P0 |
| **步骤4** | 优化体验：显示加载进度指示器 | P1 |

## 技术细节

### 为什么先不分组？
- 分组需要比较文件名，建立字典
- 第一批不分组，快速显示
- 全部扫描完再分组，性能最优

### 如何避免 UI 卡顿？
- 分批添加到 Images 集合
- 使用 DispatcherQueue 确保在 UI 线程更新
- 每批数量适中（100 个）

### 文件数阈值
- 第一批：50 个
- 后续每批：100 个
- 触发分组：全部扫描完成后

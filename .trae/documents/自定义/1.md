# 💥 真正触发点（精准定位）

你现在有三个线程在动 UI：

***

## 🧨 线程1：Rendering（每帧）

```
ImageScaleTransform.ScaleX = ...
```

***

## 🧨 线程2：异步加载完成

```
MainImage.Source = bitmap;
```

***

## 🧨 线程3：布局/动画切换

```
SwitchToViewerLayerAsync()
```

***

👉 ❗这三个**没有同步**

***

# 🚨 最致命的一点

当你执行：

```
MainImage.Source = bitmap;
```

👉 WinUI 内部会：

- &#x20;重新创建 RenderNode&#x20;
- &#x20;替换 Visual Tree&#x20;
- &#x20;更新 Transform&#x20;

***

👉 此时如果 Rendering 同时在：

```
ImageScaleTransform.ScaleX = ...
```

***

> 💥 就会打到“半更新状态的 UI 对象” → 无效指针

***

# 🧊 一句话总结根因

> ❗**你在“UI结构变更”的同时，还在“每帧修改UI属性”**

***

# ✅ 最终解决方案（这次是完整闭环）

你需要做 **两层隔离**

***

# 🥇 第一层：加载完成 → 必须暂停 Rendering

***

## ✔ 改这里（关键）

```
async void LoadHigherResolutionAsync(int targetSize)
{
    _isLoadingHighRes = true;

    var bitmap = await DecodeAsync(file, targetSize);

    _isLoadingHighRes = false;

    await DispatcherQueue.EnqueueAsync(() =>
    {
        StopPhysics();   // ❗关键：先停

        MainImage.Source = bitmap;

        _currentDecodeLevel = targetSize;

        StartPhysics();  // ❗再恢复
    });
}
```

***

👉 💥 这一行是救命的：

```
StopPhysics();
```

***

# 🥈 第二层：Rendering 避开“加载期间”

***

```
if (_isLoadingHighRes)
    return;
```

***

👉 防止：

- &#x20;解码完成前后那几帧冲突&#x20;

***

# 🥉 第三层：Transform 引用缓存（防止被替换）

***

## ❗不要每次访问 UI Tree

```
ImageScaleTransform.ScaleX = ...
```

***

## ✔ 初始化时缓存

```
_scaleTransform = ImageScaleTransform;
_translateTransform = ImageTranslateTransform;
```

***

## ✔ Rendering 用缓存

```
_scaleTransform.ScaleX = _zoomScale;
```

***

👉 避免：

> UI Tree 被换掉 → 引用失效

***

# 🧩 第四层（可选但强烈推荐）

👉 防抖加载：

```
CancellationTokenSource _cts;
```

每次新请求：

```
_cts?.Cancel();
_cts = new CancellationTokenSource();
```

***

👉 防止：

- &#x20;多个 decode 同时完成 → 连续替换 UI → 更容易炸&#x20;

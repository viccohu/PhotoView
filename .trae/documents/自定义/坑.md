# 🔥 一、线程模型类（最致命）

***

## ❌ 坑 1：在后台线程创建 UI 对象

### 你的问题：

```
new BitmapImage();
bitmap.SetSource(stream);

```

在 async 之后执行（可能在后台线程）

***

### 💥 后果：

- 直接 Win32 崩溃
- 异常码：`0xC000027B`
- 无托管异常（try/catch 捕不到）

***

### ✅ 正确原则：

```
所有 UI 对象必须在 UI线程创建

```

***

***

## ❌ 坑 2：误以为 MVVM 会“分离线程”

### 你的误区：

> 用了 MVVM → 就不是 UI线程了

***

### 实际：

```
MVVM ≠ 多线程
async 才会切线程

```

***

***

## ❌ 坑 3：WinRT 对象跨线程访问（关键坑）

### 你的问题：

```
public ImageProperties ImageProperties { get; }

```

然后在 UI 中：

```
ImageProperties.Height

```

***

### 💥 后果：

- COM 跨线程访问
- XAML 内部崩溃
- 表现为：在 `Height` 断掉

***

### ✅ 正确做法：

```
Model 中禁止保存 WinRT 对象
只保存纯数据（int / string）

```

***

***

# 💣 二、资源生命周期类（隐藏最深）

***

## ❌ 坑 4：过早释放 stream

### 你的代码：

```
using (stream)
{
    bitmap.SetSource(stream);
}

```

***

### 💥 本质问题：

```
SetSource ≠ 同步读取
WinUI 内部异步解码

```

***

### 💥 结果：

```
stream 被释放 → UI还在用 → 崩溃

```

***

### ✅ 正确：

```
禁止 using / Dispose stream（至少在 SetSource 后）

```

***

***

## ❌ 坑 5：手动 Dispose thumbnail

***

### 💥 后果：

# 🧩 十、最终工程级原则（建议你写进项目规则）

***

## 🧠 核心规则：

```
1. UI对象必须 UI线程创建
2. WinRT对象不能跨线程存活
3. stream 在 SetSource 期间不能释放
4. 所有 async 必须防复用
5. UI 只能绑定数据，不能控制数据
6. 应用退出必须停止所有任务

```

***

# 🎯 终极一句话

```
WinUI开发的本质不是写UI，
而是控制线程、生命周期和数据流。

```
